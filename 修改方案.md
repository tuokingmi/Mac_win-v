# 为 Mac-Clipboard 实现“3 分钟内下一次打开临时置顶”与 Command 多选粘贴

---

## 1. 工作目标

实现两个关联功能。

### 功能 A：3 分钟内的“下一次打开临时置顶”

用户在外部应用中复制一个或多个内容后，只要某条内容从复制时刻算起尚未满 3 分钟，它就应在剪贴板面板下一次从隐藏状态打开时，临时显示在列表最上方，甚至排在永久置顶内容之前。

这项临时提升只持续一个面板会话。该面板会话关闭后，这些内容恢复原有排序：

1. 永久置顶内容；
2. 普通历史内容。

如果某条内容复制后已经满 3 分钟，并且期间从未打开过剪贴板面板，那么以后再打开时不能再临时提升它。

### 功能 B：Command 多选与批量粘贴

剪贴板面板打开时，用户可以按住 `Command` 点击多个条目进行多选，然后按 `Enter` 或点击批量粘贴按钮，将所选内容按列表从上到下的顺序粘贴。

普通单击仍保留当前的“立即粘贴单项”行为。

---

## 2. 产品语义：必须严格遵守

以下语义是验收依据，不要自行改成其他逻辑。

### 2.1 每条复制事件独立计算 3 分钟

每个外部复制事件都有自己的：

```text
copiedAt
expiresAt = copiedAt + 3 分钟
```

仅当打开面板时满足：

```text
now < expiresAt
```

该条目才有资格进入本次临时置顶区。

在刚好达到或超过 3 分钟时，视为已过期，不再临时置顶。

不要使用“全局最后一次复制时间”统一刷新所有内容。后复制的内容不能延长较早内容的有效期。

### 2.2 多条内容同时有效时，后复制的排在更上面

示例：

```text
3:00 复制 A
3:01 复制 B
3:02 打开面板
```

显示顺序：

```text
B
A
所有永久置顶项
所有其他普通项
```

A、B 都在各自的 3 分钟窗口内，B 因为复制更晚而排在 A 上面。

### 2.3 较早内容可以过期，较晚内容仍然有效

示例：

```text
10:00 复制 A
10:08 复制 B
10:10 打开面板
```

显示顺序：

```text
B
所有永久置顶项
A 和其他普通项按普通历史顺序排列
```

A 已过期，不再越过永久置顶区；B 仍有效。

### 2.4 后复制内容不会让已过期内容重新生效

示例：

```text
10:00 复制 A
10:11 复制 B
10:12 打开面板
```

只有 B 临时置顶。A 不能因为 B 的复制而重新获得资格。

### 2.5 同一内容被再次复制时，刷新该条目自己的窗口

如果当前去重逻辑决定不创建新的历史记录，而是复用现有条目，则这次新的外部复制事件仍然必须：

- 将该条目移动到待提升队列最前；
- 将该条目的 `copiedAt` 更新为这次复制的时间；
- 重新获得从这次复制开始的 3 分钟窗口。

只刷新该条目，不刷新其他待提升条目。

### 2.6 打开一次即消费资格，不是在 3 分钟内每次打开都提升

3 分钟是“等待第一次打开”的最长时间，不是持续 3 分钟的反复提升状态。

示例：

```text
10:00 复制 A
10:01 第一次打开面板：A 临时置顶
10:02 关闭面板
10:03 再次打开面板
```

10:03 时 A 不再临时置顶，即使距离复制还不到 3 分钟。

它应恢复到：

- 永久置顶区，如果 `isPinned == true`；
- 普通历史区，如果 `isPinned == false`。

### 2.7 有效期只在“隐藏 → 显示”时判断

当面板真正从隐藏状态切换为显示状态时：

1. 清理已过期的待提升记录；
2. 将仍有效的待提升记录消费为当前会话的 active 快照；
3. 清空这些待提升记录；
4. 用 active 快照构造当前会话排序。

如果面板已经可见，再次调用 `show()` 不得重复消费下一批待提升内容。

某条内容在面板打开时仍有效，那么即使面板保持打开直到超过 3 分钟，它在本次会话中仍保持临时置顶。不要在面板可见期间通过定时器把它移走。

### 2.8 面板打开期间发生的新复制属于下一次会话

如果面板已经打开，用户又在其他位置产生了新的外部复制：

- 不要动态插入或重排当前会话的临时置顶区；
- 将它放入下一次打开的 pending 队列；
- 从它自己的复制时刻开始计算 3 分钟；
- 当前面板关闭并再次打开时，若尚未过期，再临时置顶。

如果下一次打开时已经过期，则不提升。

### 2.9 所有打开方式行为一致

以下方式打开面板时必须使用同一套消费和过期判断：

- `Option + V`；
- 菜单栏中的“Show Clipboard History”；
- 任何现有的其他 `PanelController.show()` 调用。

### 2.10 所有关闭方式都结束当前临时提升会话

以下路径都必须清理 active 快照：

- 按 `Esc`；
- 再按一次 `Option + V`；
- 点击其他应用导致 panel resign key；
- 普通单项粘贴；
- Command 多选后按 Enter；
- 点击“粘贴 N 项”按钮；
- 任何程序化关闭路径。

### 2.11 应用自身执行粘贴不算外部复制

`PasteService` 写入 `NSPasteboard.general` 时，不得：

- 新建历史记录；
- 刷新某条记录的 3 分钟窗口；
- 将内容加入下一次临时置顶队列。

必须可靠地区分外部复制与应用内部写入。

### 2.12 永久置顶状态与临时提升状态相互独立

临时提升不能修改：

- `ClipboardItem.isPinned`；
- `ClipboardItem.createdAt`，除非当前捕获流程本来就要创建或更新一条新记录；
- 用户的永久置顶设置。

一个永久置顶条目因新复制而临时提升时：

```text
本次打开：临时提升区
关闭后下次打开：永久置顶区
```

### 2.13 pending 状态只保存在内存中

不要把一次性临时提升状态写入 SwiftData。

应用退出或崩溃后，所有尚未消费的临时提升资格可以丢失。不要为此增加模型迁移。

## 3. 明确示例

假设永久置顶项为 `P1、P2`，普通历史项为 `U1`。

### 示例 1：连续复制并及时打开

```text
10:00 复制 A
10:02 复制 B
10:03 打开
```

顺序：

```text
B、A、P1、P2、U1
```

关闭后再次打开：

```text
P1、P2、B、A、U1
```

### 示例 2：一条过期、一条有效

```text
10:00 复制 A
10:08 复制 B
10:11 打开
```

顺序：

```text
B、P1、P2、A、U1
```

### 示例 3：全部过期

```text
10:00 复制 A
10:05 复制 B
10:16 打开
```

A、B 都不临时提升，按永久置顶和普通历史规则排序。

### 示例 4：当前会话期间复制

```text
10:00 复制 A
10:01 打开，A 临时置顶
10:02 面板仍打开时复制 B
10:03 关闭
10:04 再次打开
```

10:04 的顺序：

```text
B、P1、P2、A、U1
```

A 已经在上一会话消费；B 属于下一会话且仍有效。

### 示例 5：当前会话期间复制但下次打开太晚

```text
10:00 打开面板
10:01 面板打开期间复制 B
10:02 关闭
10:12 再次打开
```

B 已满 3 分钟，不临时提升。

### 示例 6：重复复制同一条目

```text
10:00 复制 A
10:09 再次复制 A
10:15 打开
```

如果去重后仍是同一个 `ClipboardItem.id`，A 仍应临时提升，因为第二次复制把 A 自己的有效期刷新到 10:19。

## 4. 推荐状态模型

请实现两个不同生命周期的状态。

```text
外部复制事件
    ↓
pendingNextOpenPromotions
    ↓ 面板从隐藏切换为显示，并在此刻清理过期项
activeSessionPromotions
    ↓ 面板以任意方式关闭
清空 active，恢复永久置顶/普通历史排序
```

建议定义一个非持久化值类型。可以放在 `ClipboardStore.swift` 顶层，避免无必要地新增工程文件：

```swift
struct NextOpenPromotion: Equatable, Sendable {
    static let validityInterval: TimeInterval = 10 * 60

    let itemID: UUID
    let copiedAt: Date

    var expiresAt: Date {
        copiedAt.addingTimeInterval(Self.validityInterval)
    }

    func isEligible(at now: Date) -> Bool {
        now < expiresAt
    }
}
```

允许采用等价设计，但必须同时保留：

- 条目 ID；
- 本次外部复制时间；
- 可确定的独立过期时间；
- 后复制优先的稳定顺序。

不要只保存 `[UUID]`，否则无法正确处理各条记录独立过期。

## 5. 文件级实施方案

## 5.1 `ClipboardMenuBar/Services/ClipboardStore.swift`

### 5.1.1 增加 pending 队列

增加：

```swift
private var pendingNextOpenPromotions: [NextOpenPromotion] = []
```

实现统一 upsert 方法：

```swift
private func enqueueNextOpenPromotion(
    itemID: UUID,
    copiedAt: Date
)
```

要求：

1. 先清理在 `copiedAt` 时刻已经过期的 pending 项；
2. 同一个 `itemID` 最多保留一条 pending 记录；
3. 如果同一 ID 再次复制，用新的 `copiedAt` 替换旧记录；
4. 所有 pending 项始终按 `copiedAt` 倒序排列；
5. 后复制的条目位于数组前面。

建议实现：

```swift
private func enqueueNextOpenPromotion(
    itemID: UUID,
    copiedAt: Date
) {
    pendingNextOpenPromotions.removeAll {
        !$0.isEligible(at: copiedAt)
    }

    pendingNextOpenPromotions.removeAll {
        $0.itemID == itemID
    }

    pendingNextOpenPromotions.append(
        NextOpenPromotion(
            itemID: itemID,
            copiedAt: copiedAt
        )
    )

    pendingNextOpenPromotions.sort {
        $0.copiedAt > $1.copiedAt
    }
}
```

### 5.1.2 增加消费方法

实现：

```swift
func consumeEligibleNextOpenPromotions(
    at now: Date = .now
) -> [NextOpenPromotion]
```

要求：

1. 删除 `now >= expiresAt` 的记录；
2. 返回仍有效的记录，保持“最新复制在前”；
3. 返回后从 pending 中清除这些记录；
4. 过期记录也永久丢弃；
5. 方法调用后 pending 应为空，除非实现中存在明确的并发隔离逻辑。

推荐：

```swift
func consumeEligibleNextOpenPromotions(
    at now: Date = .now
) -> [NextOpenPromotion] {
    let eligible = pendingNextOpenPromotions
        .filter { $0.isEligible(at: now) }
        .sorted { $0.copiedAt > $1.copiedAt }

    pendingNextOpenPromotions.removeAll()
    return eligible
}
```

由于整个 Store 是 `@MainActor`，无需引入锁。

### 5.1.3 修改列表排序接口

将：

```swift
func fetchItems() -> [ClipboardItem]
```

改为兼容旧调用的接口：

```swift
func fetchItems(
    promoting promotions: [NextOpenPromotion] = []
) -> [ClipboardItem]
```

排序必须为：

```text
promotions 指定的有效 active 顺序
+ 其余永久置顶项
+ 其余普通项
```

示例实现：

```swift
func fetchItems(
    promoting promotions: [NextOpenPromotion] = []
) -> [ClipboardItem] {
    let descriptor = FetchDescriptor<ClipboardItem>(
        sortBy: [
            SortDescriptor(
                \ClipboardItem.createdAt,
                order: .reverse
            )
        ]
    )

    let all = (try? modelContext.fetch(descriptor)) ?? []
    let itemsByID = Dictionary(
        uniqueKeysWithValues: all.map { ($0.id, $0) }
    )

    let promoted = promotions.compactMap {
        itemsByID[$0.itemID]
    }

    let promotedIDs = Set(promoted.map(\.id))

    let pinned = all.filter {
        $0.isPinned && !promotedIDs.contains($0.id)
    }

    let unpinned = all.filter {
        !$0.isPinned && !promotedIDs.contains($0.id)
    }

    return promoted + pinned + unpinned
}
```

注意：

- active 的顺序必须来自 `promotions`；
- 不要把 active 再按模型 `createdAt` 排序；
- 同一个条目不得重复出现；
- 已过期记录不应进入 active，过期判断在消费时完成。

### 5.1.4 重构捕获 reservation，确保图片异步处理也使用真实复制时间

当前图片在后台处理完成后才保存。如果直接用处理完成时间，会破坏 3 分钟判断和多图片复制顺序。

建议增加：

```swift
struct ClipboardCaptureToken: Sendable, Equatable {
    let id: UUID
    let signature: String
}

private struct InFlightCapture {
    let token: ClipboardCaptureToken
    var copiedAt: Date
}
```

Store 保存：

```swift
private var inFlightCaptures: [String: InFlightCapture] = [:]
```

提供：

```swift
enum ClipboardCaptureReservation {
    case existing(UUID)
    case alreadyInFlight(UUID)
    case new(ClipboardCaptureToken)
}
```

以及：

```swift
func reserveExternalCapture(
    signature: String,
    copiedAt: Date = .now
) -> ClipboardCaptureReservation
```

逻辑：

1. 如果相同 signature 正在异步处理中：
   - 复用正在处理的 item ID；
   - 把该 in-flight state 的 `copiedAt` 更新为本次复制时间；
   - 刷新该 ID 的 pending 3 分钟窗口；
   - 不启动第二个相同图片任务；
   - 返回 `.alreadyInFlight(id)`。

2. 如果当前去重规则判断已有可复用历史项：
   - 不创建重复历史；
   - 用本次复制时间 enqueue；
   - 返回 `.existing(item.id)`。

3. 否则：
   - 创建稳定 UUID；
   - 保存 in-flight state；
   - 立即 enqueue，不能等图片处理完成后再入队；
   - 返回 `.new(token)`。

提交接口建议：

```swift
func commitText(
    _ text: String,
    token: ClipboardCaptureToken
)

func commitImage(
    payload: StoredImagePayload,
    token: ClipboardCaptureToken
)

func cancelCapture(
    _ token: ClipboardCaptureToken
)
```

提交时从 `inFlightCaptures` 读取最新 `copiedAt`，使用它创建模型：

```swift
let item = ClipboardItem(
    id: token.id,
    createdAt: inFlight.copiedAt,
    kind: ...,
    pasteboardSignature: token.signature
)
```

这样同一图片在异步处理期间被再次复制时，最终模型时间和临时提升窗口都反映最后一次复制事件。

提交或取消后移除对应 in-flight state。

如果图片处理失败：

- 删除临时图片文件；
- 移除 in-flight state；
- 移除该 ID 尚未消费的 pending promotion；
- 触发必要的 UI 更新；
- 不留下不可显示的幽灵条目。

### 5.1.5 处理“面板已消费，但图片还未完成”的边界

面板打开时可能消费了某个图片 ID，但图片仍在后台处理，因此本次会话还没有真正显示它。

增加：

```swift
func finishPresentationSession(
    promotions: [NextOpenPromotion],
    at now: Date = .now
)
```

仅对以下记录重新放回 pending：

- 仍处于 `inFlightCaptures`；
- 在 `now` 时仍未过期；
- 本次会话中尚未完成持久化，因而可能没有展示。

重新入队时必须保留原 promotion 的 `copiedAt`，不能把关闭时间当成新的复制时间，也不能延长 3 分钟窗口。

如果同一 ID 在面板打开期间又产生了更新的 pending promotion，不得用旧 active promotion 覆盖更新的 `copiedAt`。upsert 时保留时间较新的记录。

这项重新入队只是图片异步尚未完成时的容错，不适用于已经成功展示的普通条目。

### 5.1.6 删除和清空时同步清理状态

修改：

- `delete(_:)`：从 pending 和 in-flight 关联状态中清理该 ID；
- `clearAll()`：清理被删除普通项对应的 pending；
- 永久置顶项未被 `clearAll()` 删除时，其 pending 可以保留；
- 图片任务完成时，如果对应 reservation 已被取消或删除，应删除刚生成的图片文件，不得重新插入历史；
- `togglePin(_:)` 不修改 promotion 的 `copiedAt` 或有效期。

## 5.2 `ClipboardMenuBar/Services/ClipboardMonitor.swift`

### 5.2.1 使用实际检测到变化的时间

在发现新的外部 pasteboard change 时立即记录：

```swift
let copiedAt = Date()
```

该时间必须传入 `reserveExternalCapture`。不要在图片后台处理完成时才生成时间。

### 5.2.2 先 reserve，再保存或启动图片任务

文本流程：

```swift
let signature = digest(...)
let copiedAt = Date()

switch clipboardStore.reserveExternalCapture(
    signature: signature,
    copiedAt: copiedAt
) {
case .existing, .alreadyInFlight:
    return

case .new(let token):
    clipboardStore.commitText(
        text,
        token: token
    )
}
```

图片流程：

1. 计算 signature；
2. 调用 reservation；
3. `.new(token)` 时启动后台图片存储；
4. 完成后回主线程 `commitImage`；
5. 失败时回主线程 `cancelCapture`；
6. `.existing` 或 `.alreadyInFlight` 不再启动重复处理任务。

原有单一 `processingImageSignature` 如与新的 in-flight 字典重复，应删除或改造，避免两套状态冲突。

### 5.2.3 用 pasteboard changeCount 抑制内部写入

现有单个 `suppressedSignature` 不足以可靠覆盖批量连续粘贴。

在 `ClipboardStore` 中增加：

```swift
private var internalPasteboardChangeCounts: Set<Int> = []

func suppressCapture(
    changeCount: Int
)

func consumeCaptureSuppression(
    changeCount: Int
) -> Bool
```

Monitor 在读取内容前：

```swift
let currentChangeCount = pasteboard.changeCount
guard currentChangeCount != lastChangeCount else {
    return
}

lastChangeCount = currentChangeCount

guard !clipboardStore.consumeCaptureSuppression(
    changeCount: currentChangeCount
) else {
    return
}
```

要求：

- 本应用写入 pasteboard 后记录写入完成时的最终 changeCount；
- Monitor 命中该 changeCount 时直接返回；
- 内部写入不进入 reservation；
- 内部写入不刷新 3 分钟窗口；
- 清理小于等于当前观察值的过期 suppression count，避免集合无限增长；
- 外部后续写入的新 changeCount 不得被误抑制。

完成后删除旧的 signature suppression 职责：

- `suppressedSignature`；
- `suppressNextCapture(signature:)`；
- `shouldSave` 中用于识别应用内部粘贴的分支。

signature 仍可以用于历史去重。

### 5.3.2 只在真正打开新会话时消费

修改 `show()`：

```swift
func show() {
    if panel.isVisible {
        panel.makeKey()
        return
    }

    appServices.refreshSystemState()

    pasteTarget = pasteService.captureTarget(
        for: NSWorkspace.shared.frontmostApplication

    activeSessionPromotions =
        clipboardStore.consumeEligibleNextOpenPromotions(

        )

    updateContent()
    positionPanel()
    panel.orderFrontRegardless()
    panel.makeKey()
}
```

要求：

- 消费和过期判断发生在 `hidden → visible`；
- 已显示时不得消费下一批；
- 当前 active 快照在本会话中保持不变；
- 面板打开期间产生的新 pending 不加入当前 active。

### 5.3.3 统一结束会话

增加：

```swift
private func finishPresentationSession() {
    clipboardStore.finishPresentationSession(
        promotions: activeSessionPromotions,
        at: .now
    )

    activeSessionPromotions.removeAll()

    pasteTarget = PasteTarget(
        application: nil,
        focusedElement: nil
    )
}
```

`hide()`：

```swift
func hide() {
    finishPresentationSession()
    panel.hideImmediately()
}
```

### 5.3.4 所有粘贴路径也必须统一收尾

不要让 `PasteService` 直接隐藏 panel。

建议：

```swift
@discardableResult
func paste(
    _ items: [ClipboardItem]
) -> Bool {
    guard let plan = pasteService.makePlan(
        for: items,
        using: clipboardStore
    ) else {
        return false
    }

    let target = pasteTarget

    finishPresentationSession()
    panel.hideImmediately()

    pasteService.execute(
        plan,
        using: clipboardStore,
        target: target
    )

    return true
}

@discardableResult
func paste(
    _ item: ClipboardItem
) -> Bool {
    paste([item])
}
```

先创建 `PastePlan` 并验证所有图片可读取，再关闭 panel，避免面板消失但粘贴失败。

### 5.3.5 把 active 快照传给列表

```swift
private func updateContent() {
    let rootView = ClipboardListView(
        clipboardStore: clipboardStore,
        panelController: self,
        activePromotions: activeSessionPromotions
    )

    panel.contentView = NSHostingView(
        rootView: rootView
    )
}
```

`makePanel()` 初始使用空数组。

```swift
private func handleRowClick(
    _ item: ClipboardItem

    let flags =
        NSApp.currentEvent?.modifierFlags
        ?? NSEvent.modifierFlags

    focusedItemID = item.id

    if flags.contains(.command) {
        if selectedItemIDs.contains(item.id) {
            selectedItemIDs.remove(item.id)
        } else {
            selectedItemIDs.insert(item.id)

        return
    }

    selectedItemIDs.removeAll()
    _ = panelController.paste(item)
}
```

## 5.6 `ClipboardMenuBar/Services/PasteService.swift`

### 5.6.1 移除面板控制职责

从 `PasteService` 删除：

- `ClipboardPanel` 参数；
- `panel.hideImmediately()`；
- 其他直接关闭 UI 的代码。

`PasteService` 只负责：

1. 创建可执行的粘贴计划；
2. 写入 pasteboard；
3. 记录内部 changeCount；
4. 激活目标应用；
5. 执行 AX 插入或 Command+V。

### 5.6.2 增加 PastePlan

建议：

```swift
enum PastePayload {
    case text(String)
    case image(NSImage)
}

enum PastePlan {
    case text(String)
    case sequential([PastePayload])
}
```

实现：

```swift
func makePlan(
    for items: [ClipboardItem],
    using store: ClipboardStore
) -> PastePlan?
```

规则：

1. 空数组返回 `nil`；
2. 全部为文本时，按列表顺序用一个换行符连接；
3. 只要包含图片，生成顺序 payload；
4. 任意图片无法读取时返回 `nil`；
5. 不要在执行到一半时才发现资源缺失。

### 5.6.3 纯文本批量粘贴

多个文本：

```swift
let combined = texts.joined(
    separator: "\n"
)
```

只执行一次 pasteboard 写入和一次粘贴动作。

保持现有优先策略：

1. 有辅助功能权限且 AX 直接插入成功：直接插入；
2. 否则有权限时发送一次 Command+V；
3. 无权限时激活目标应用，并把合并文本留在系统剪贴板供用户手动粘贴。

### 5.6.4 图片或混合内容

有辅助功能权限时，按 payload 顺序逐项：

```text
激活目标应用
确认应用 active
写入第 1 项
记录最终 changeCount
发送 Command+V
等待约 180–250 ms
写入第 2 项
记录最终 changeCount
发送 Command+V
……
```

集中定义间隔，例如：

```swift
private let multiPasteInterval:
    UInt64 = 220_000_000
```

不要把魔法延时散落在多个函数中。

混合批次统一使用顺序 Command+V，不要对其中的文本走 AX 直接插入，否则富文本编辑器中的插入位置可能与图片粘贴不同步。

无辅助功能权限时：

- 将所有 payload 尽可能一次性写入 pasteboard；
- 记录最终 changeCount；
- 激活目标应用；
- 不伪造键盘事件；
- 用户可手动 Command+V。

### 5.6.5 每次内部写入后登记最终 changeCount

统一封装：

```swift
private func markInternalWrite(
    pasteboard: NSPasteboard,
    store: ClipboardStore
) {
    store.suppressCapture(
        changeCount: pasteboard.changeCount
    )
}
```

必须在内容写入完成后登记，不能只在 `clearContents()` 后登记，因为清空和写入都可能改变 changeCount。

完成后 `PasteService` 不再需要为了 suppression 计算 signature；如果没有其他用途，可以移除该文件中的 CryptoKit import。
